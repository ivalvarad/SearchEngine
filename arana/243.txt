<p><b>Portals</b> is a low-level network <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a> for high-performance networking on <a href="/wiki/High-performance_computing" title="High-performance computing" class="mw-redirect">high-performance computing</a> systems developed by <a href="/wiki/Sandia_National_Laboratories" title="Sandia National Laboratories">Sandia National Laboratories</a> and the <a href="/wiki/University_of_New_Mexico" title="University of New Mexico">University of New Mexico</a>. Portals is currently the lowest-level network programming interface on the commercially successful XT line of supercomputers from <a href="/wiki/Cray" title="Cray">Cray</a>.</p>

<h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portals_network_programming_application_programming_interface&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Portals is based on the concept of elementary building blocks that can be combined to support a wide variety of upper-level network transport semantics. Portals provides one-sided data movement operations, but unlike other one-sided programming interfaces, the target of a remote operation is not a <a href="/wiki/Virtual_address" title="Virtual address" class="mw-redirect">virtual address</a>. Instead, the ultimate destination in memory of an incoming message is determined at the receiver by comparing contents of the message header with the contents of structures at the destination. This flexibility allows for efficient implementations of both one-sided and two-sided communications. In particular, Portals is aimed at providing the fundamental operations necessary to support a high-performance and scalable implementation of the <a href="/wiki/Message_Passing_Interface" title="Message Passing Interface">Message Passing Interface</a> (MPI) standard.<sup id="cite_ref-brightwell-1996_1-0" class="reference"><a href="#cite_note-brightwell-1996-1"><span>[</span>1<span>]</span></a></sup> It was also used as the initial network transport layer for the <a href="/wiki/Lustre_(file_system)" title="Lustre (file system)">Lustre</a> file system.</p>
<h2><span class="mw-headline" id="Early_years">Early years</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portals_network_programming_application_programming_interface&amp;action=edit&amp;section=2" title="Edit section: Early years">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Portals began in the early 1990s as an extension to the <a href="/wiki/NCUBE" title="NCUBE">nX</a> message passing system used in the <a href="/wiki/SUNMOS" title="SUNMOS">SUNMOS</a> and <a href="/w/index.php?title=PUMA_Cluster_Super_Computer&amp;action=edit&amp;redlink=1" class="new" title="PUMA Cluster Super Computer (page does not exist)">Puma</a> operating system. It was first implemented for the <a href="/wiki/Intel_Paragon" title="Intel Paragon">Intel Paragon</a> at Sandia, and later ported to the <a href="/w/index.php?title=Intel_Teraflops&amp;action=edit&amp;redlink=1" class="new" title="Intel Teraflops (page does not exist)">Intel TeraFLOPS</a> machine named <a href="/wiki/ASCI_Red" title="ASCI Red">ASCI Red</a>.<sup id="cite_ref-brightwell-1996_1-1" class="reference"><a href="#cite_note-brightwell-1996-1"><span>[</span>1<span>]</span></a></sup> There were four building blocks in the first version of Portals: the single block, the dynamic block, the independent block and the combined block. All incoming messages would first pass through a match-list that allowed individual portals to respond to specific groups, ranks, and a set of user specified match-bits.<sup id="cite_ref-brightwell-1996_1-2" class="reference"><a href="#cite_note-brightwell-1996-1"><span>[</span>1<span>]</span></a></sup></p>
<h2><span class="mw-headline" id="Portals_3">Portals 3</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portals_network_programming_application_programming_interface&amp;action=edit&amp;section=3" title="Edit section: Portals 3">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The Portals concept continued to evolve over successive generations of lightweight kernels and massively parallel systems. In 1999, an operational programming interface was given to Portals so that it could be implemented for intelligent or programmable network interfaces outside of a lightweight kernel environment.<sup id="cite_ref-portals3.0_2-0" class="reference"><a href="#cite_note-portals3.0-2"><span>[</span>2<span>]</span></a></sup> This standard was designed for systems where the work required to prepare, transmit, and deliver messages is longer than the round-trip to the Portals data structures. For example, in modern systems, this work is dominated by the round-trip through the IO bus to the network interface. The standard has been revised since the initial release to make it more suited for modern high performance, massively parallel computers.<sup id="cite_ref-portals3.3_3-0" class="reference"><a href="#cite_note-portals3.3-3"><span>[</span>3<span>]</span></a></sup> The MPI library was ported from the <a href="/wiki/Retronym" title="Retronym">retronymed</a> Portals 2 to the new Portals 3.0.<sup id="cite_ref-brightwell-2002_4-0" class="reference"><a href="#cite_note-brightwell-2002-4"><span>[</span>4<span>]</span></a></sup></p>
<p>The Portals 3 specification has been implemented several times, first on <a href="/wiki/ASCI_Red" title="ASCI Red">ASCI Red</a>, then on <a href="/w/index.php?title=CPlant&amp;action=edit&amp;redlink=1" class="new" title="CPlant (page does not exist)">CPlant</a> over <a href="/wiki/Myrinet" title="Myrinet">Myrinet</a>,<sup id="cite_ref-pundit-2001_5-0" class="reference"><a href="#cite_note-pundit-2001-5"><span>[</span>5<span>]</span></a></sup> <a href="/wiki/Linux" title="Linux">Linux</a> and the <a href="/wiki/Cray_XT3" title="Cray XT3">Cray XT</a> family.<sup id="cite_ref-pedretti-2005_6-0" class="reference"><a href="#cite_note-pedretti-2005-6"><span>[</span>6<span>]</span></a></sup></p>
<h2><span class="mw-headline" id="Portals_4">Portals 4</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Portals_network_programming_application_programming_interface&amp;action=edit&amp;section=4" title="Edit section: Portals 4">edit</a><span class="mw-editsection-bracket">]</span></span></h2>
<p>In light of emerging <a href="/wiki/Partitioned_global_address_space" title="Partitioned global address space">partitioned global address space</a> (PGAS) languages, several new features have been added to the Portals API as part of Portals 4. Portals 4 also made several changes to improve the interaction between the processor and network interface (NIC) for implementations that provide offload. Finally, an option to support a form of flow-control was added to Portals 4.<sup id="cite_ref-portals4.0_7-0" class="reference"><a href="#cite_note-portals4.0-7"><span>[</span>7<span>]</span></a></sup></p>
